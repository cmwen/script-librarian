# Design Direction

## Product Identity

Script Librarian is a quiet, fast, trustworthy local tool. It should feel closer
to a well-kept workshop drawer than a chat product. The product can use AI, but
the user's script library is the main object.

Working product phrase:

```text
Your local reusable automation library, shared with your AI assistants.
```

## Interaction Model

The app has three surfaces:

- Human CLI
- Managed script registry
- Agent/MCP interface

The human CLI should be useful without AI. The LLM generator should be an
optional accelerator. MCP should allow external agents to participate in the
same library.

## Information Architecture

Primary objects:

- Script
- Script metadata
- Managed script home
- LLM provider
- Git history
- MCP tool surface

Primary script fields:

- command name
- display name
- description
- usage
- tags
- aliases
- runtime
- safety level
- dependencies
- examples
- path
- version/history

## CLI Design

The default `msl` command should open the fastest possible picker. It should
avoid loading generation, MCP, web, or provider code until needed.

Primary commands:

```bash
msl
msl init
msl search <query>
msl info <script>
msl run <script> -- <args>
msl new <prompt>
msl improve <script> <prompt>
msl scripts status
msl scripts log <script>
msl llm configure
msl mcp
```

The CLI should make safety and provenance visible:

```text
Safety: writes-project
Generated by: ollama/gemma
Last changed: 2026-07-02
```

## Script Metadata Design

The simplest portable format is top-of-file comments:

```bash
#!/usr/bin/env bash
# msl:name Repo Cleanup
# msl:description Remove merged local git branches after confirmation.
# msl:usage repo-cleanup [--dry-run]
# msl:tags git, cleanup, branches
# msl:runtime bash
# msl:safety writes-project
# msl:deps git
# msl:alias rc
```

This keeps scripts self-contained and easy to inspect. A richer sidecar file may
be added later if comment metadata becomes too limited.

## Generation Design

Generation should be constrained. The model should produce one script that fits
the script shape, not a general project patch.

Generation pipeline:

```text
User prompt
  -> search similar scripts
  -> assemble generation context
  -> call provider adapter
  -> parse generated script
  -> validate metadata and safety
  -> show diff
  -> save and chmod
  -> commit if enabled
```

The LLM prompt should strongly prefer:

- bash for simple local automation
- `--help`
- `--dry-run` for risky operations
- conservative defaults
- clear error messages
- dependency checks
- metadata completeness

## Provider Adapter Design

Provider adapters should be small and replaceable. The product should support
local and hosted models without turning into a coding-agent orchestrator.

Initial adapter interface:

```text
generateScript(request) -> generated script text
```

Adapter inputs:

- user request
- script shape
- similar scripts
- preferred runtime
- safety policy
- output constraints

Likely adapters:

- OpenAI-compatible endpoint
- Ollama
- LM Studio
- Anthropic
- OpenAI
- custom command

## MCP Design

MCP is for outside agents. The built-in generator should not depend on MCP, but
both should use the same registry, metadata parser, validator, and git workflow.

Recommended MCP tools:

```text
list_scripts
search_scripts
get_script
get_script_shape
validate_script
save_script
update_script
```

Execution through MCP should be treated as sensitive. If included, it should be
dry-run first and require explicit user confirmation for real execution.

## Storage Design

The managed script home should be user-owned and easy to back up.

Possible defaults:

```text
~/scripts
~/.local/share/script-librarian/scripts
```

Version control should be plain git. This avoids inventing a custom history
system and gives users rollback, sync, and review with familiar tools.

## Installation Direction

The installation path is undecided. The product should eventually support:

- one-command install for the CLI
- shell PATH setup guidance
- optional MCP configuration snippet
- optional LLM provider setup
- no required cloud account

## Technology Direction

The implementation stack is intentionally undecided.

Evaluation criteria:

- fast startup
- small executable or small install footprint
- easy cross-platform packaging
- strong filesystem/process APIs
- pleasant TUI development
- simple MCP server support
- provider adapters without heavy dependencies

Candidate directions:

- Go with a native TUI framework
- Rust with a native TUI framework
- bundled JavaScript with external runtime
- other small native toolchain

The product design should not depend on this choice yet.

## Open Design Questions

- Should the product name be Script Librarian, while the command remains `msl`?
- Should generated scripts be saved immediately after review, or first staged as
  drafts?
- Should the library support folders/categories, or rely on tags and search?
- Should script execution history be recorded locally?
- Should MCP save operations commit automatically or stage for user review?
- How visible should LLM provider configuration be during normal use?
